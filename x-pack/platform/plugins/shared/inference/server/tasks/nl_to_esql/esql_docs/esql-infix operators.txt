# INFIX OPERATORS

Infix operators provide convenient syntax for common operations such as type casting, set membership, pattern matching, and full-text search.

## Syntax

- Cast: `expression::type`
- IN: `expression IN (value1, value2, ...)`
- LIKE: `expression LIKE pattern` or `expression LIKE (pattern1, pattern2, ...)`
- RLIKE: `expression RLIKE pattern` or `expression RLIKE (pattern1, pattern2, ...)`
- Match: `field:query_string`

### Parameters

#### Cast (`::`)

The `::` operator provides an alternative syntax to the TO_<type> conversion functions, allowing you to cast values to a different type.

#### IN

The `IN` operator tests whether a field or expression equals any element in a list of literals, fields, or expressions.

#### LIKE

The `LIKE` operator filters data based on string patterns using wildcards. The left-hand side is typically a field or expression, and the right-hand side is the pattern or list of patterns. Supported wildcards:
- `*` matches zero or more characters.
- `?` matches one character.

LIKE is case-sensitive and matches the exact string indexed. Escaping special characters such as `*` and `.` requires a backslash (`\`), which itself must be escaped in string literals. Triple-quoted strings (`"""`) can be used to reduce escaping overhead. Both single patterns and lists of patterns are supported.

#### RLIKE

The `RLIKE` operator filters data based on string patterns using regular expressions. The left-hand side is typically a field or expression, and the right-hand side is the pattern or list of patterns. Pattern matching is case-sensitive and matches the exact string indexed. Escaping special characters (e.g., `.`, `*`, `(`) requires a backslash (`\`), which itself must be escaped in string literals. Triple-quoted strings (`"""`) can be used to reduce escaping overhead. Both single patterns and lists of patterns are supported.

#### Match operator (`:`)

The match operator (`:`) performs full-text search and filters rows that match a given query string. It is equivalent to the match function and can be used on `text`, `keyword`, and `semantic_text` fields. If a single value in a multi-valued column matches, the expression evaluates to `TRUE`.

## Examples

Cast a string to integer, then back to string, and finally to version:
This converts the string "0" to an integer, adds 1, converts the result back to a string, concatenates it with ".2.3", and casts the final value to a version type.
```esql
ROW ver = CONCAT(("0"::INT + 1)::STRING, ".2.3")::VERSION
```

Test if an expression result is in a list of values:
Checks whether the result of `c-a` is present in the list `(3, b / 2, a)`.
```esql
ROW a = 1, b = 4, c = 3
| WHERE c-a IN (3, b / 2, a)
```

Filter first names matching a pattern with wildcards:
Selects employees whose first name matches the pattern `?b*`, where `?` is any single character and `*` is any sequence of characters.
```esql
FROM employees
| WHERE first_name LIKE """?b*"""
| KEEP first_name, last_name
```

Escape special characters in LIKE patterns:
Filters rows where the message field contains the literal string "foo * bar", escaping the asterisk.
```esql
ROW message = "foo * bar"
| WHERE message LIKE "foo \\* bar"
```

Reduce escaping with triple-quoted strings:
Matches messages containing the literal string "foo * bar" using triple-quoted strings to simplify escaping.
```esql
ROW message = "foo * bar"
| WHERE message LIKE """foo \* bar"""
```

Match any of multiple patterns:
Returns rows where the message matches either the pattern "foo*" or "bar?".
```esql
ROW message = "foobar"
| WHERE message like ("foo*", "bar?")
```

Use a REST query placeholder for the pattern:
Filters employees by first name using a pattern provided as a query parameter.
```esql
FROM employees
| WHERE first_name LIKE ?pattern
| KEEP first_name, last_name
```

Filter first names matching a regular expression:
Selects employees whose first name matches the regular expression `.leja.*`.
```esql
FROM employees
| WHERE first_name RLIKE """.leja.*"""
| KEEP first_name, last_name
```

Escape special characters in RLIKE patterns:
Filters rows where the message field contains the literal string "foo ( bar", escaping the parenthesis.
```esql
ROW message = "foo ( bar"
| WHERE message RLIKE "foo \\( bar"
```

Reduce escaping with triple-quoted strings in RLIKE:
Matches messages containing the literal string "foo ( bar" using triple-quoted strings to simplify escaping.
```esql
ROW message = "foo ( bar"
| WHERE message RLIKE """foo \( bar"""
```

Match any of multiple regular expressions:
Returns rows where the message matches either the regular expression "foo.*" or "bar.".
```esql
ROW message = "foobar"
| WHERE message RLIKE ("foo.*", "bar.")
```

Use a REST query placeholder for the RLIKE pattern:
Filters employees by first name using a regular expression provided as a query parameter.
```esql
FROM employees
| WHERE first_name RLIKE ?pattern
| KEEP first_name, last_name
```

Full-text search on a text field using the match operator:
Filters books where the author field matches the full-text search query "Faulkner".
```esql
FROM books
| WHERE author:"Faulkner"
```

Full-text search on a keyword field:
Selects employees whose job_positions field matches the full-text search query "Internship" and keeps their employee number and job positions.
```esql
FROM employees
| WHERE job_positions:"Internship"
| KEEP emp_no, job_positions
```

Semantic search using the match operator and sorting by relevance:
Retrieves books whose semantic_title matches "Shakespeare", sorts them by relevance score, and keeps the score and title.
```esql
FROM books METADATA _score
| WHERE semantic_title:"Shakespeare"
| SORT _score DESC
| KEEP _score, semantic_title
```
